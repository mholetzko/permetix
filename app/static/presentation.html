Automotive Model

Automotive Development Journey (Edge/IoT)

Customer feedback is far from the developer; the journey classically ends at customer release
  • Plan ‚Üí Design ‚Üí Build (HW/SW, safety cases, determinism)
  • Integrate & Validate (HIL/SIL, homologation, long cycles)
  • Release to customer ("must be perfect" mindset)
  • Maintenance mode (very rare changes, long lead times)
  • Feedback distance: signals arrive late via tickets/reports, not shared telemetry

Inspired by developer journey mapping and feedback-loop thinking [1]

Cloud Model

Cloud Development Journey

Product team owns build ‚Üí run with shared observability and short feedback cycles
  • Plan ¬∑ Design (product backlog, SLOs defined)
  • Build (CI, tests, feature flags)
  • Deploy (CD, fast rollback)
  • Operate (SLOs/alerts, dashboards, on‚Äëcall)
  • Feedback is near: live telemetry to the team (logs, metrics, traces)
Cloud DevOps

Direct Telemetry Access

Cloud teams have direct, immediate access to telemetry
  • Alert fires ‚Üí on-call paged
  • Open dashboard ‚Üí see exact failure
  • Filter logs by version, user, time
  • Correlate metrics ‚Üí code ‚Üí commit
  • Minutes to triage and fix
Why cloud is different

What is different in the cloud?

  • Process + technology ‚Äì it's the operating model and the tech, not just tech.
  • Release is not the end ‚Äì shipping isn't enough; running it is part of the product.
  • Fewer handoffs ‚Äì fewer tickets and hands; the problem stays clearer end‚Äëto‚Äëend.
  • Shared observability stack ‚Äì L1 to developer sees the same data (logs, metrics, traces).
  • SLO‚Äëbased alerts ‚Äì proactive and metrics‚Äëdriven, not only reactive to customer reports.
  • Easy Linux‚Äëbased hardware abstraction ‚Äì standardized platform enables portability (this is where we want to be with MB.OS).

Bridging the Gap

The thin line: Cloud benefits vs. Automotive realities

Not everything that glitters is gold ‚Äî different goals and constraints

Automotive (Edge/IoT)

Safety & regulation first ‚Äî homologation, determinism, long validation

Expectation of perfection at release ‚Äî customer-ready, very rare changes

Feedback distance ‚Äî Strac/Jira/‚Ä¶ instead of shared live telemetry

Cloud DevOps

Operate as part of the product ‚Äî team owns build ‚Üí run

Near feedback ‚Äî logs/metrics/traces + SLO alerts

Frequent change ‚Äî CI/CD, flags, rollback in minutes

Cloud practices improve detection and ownership; Automotive constraints protect safety and brand ‚Äî the art is choosing what to adopt without breaking what matters.
Cloud DevOps

It's Not About the Tools

It is not a "tool" which enables CI/CD and cloud, it is the working model itself
  • CI/CD pipelines are tools, but DevOps is the operating model
  • The model: same team owns build ‚Üí run ‚Üí feedback
  • Shared observability enables this model at scale
  • Tools support the model; the model enables the outcomes
